== What is Axiom.jl?

Axiom.jl is a small, pure-Julia neural-network library whose distinguishing
feature is a property-checking layer on top of inference: you build a model from
`Dense`/`Conv2d`/normalisation/pooling layers, run a forward pass, and then call
`verify(model; properties, data)` to assert runtime invariants such as "outputs
are finite" or "outputs are valid probabilities". It also provides an `@axiom`
model-definition macro, `@ensure`/`@prove` assertion macros, a Zygote-backed
autograd path, basic optimisers and losses, normalising-flow / invertible
layers, a JSON model-packaging format, HTTP/GraphQL/gRPC inference servers, a
PyTorch-descriptor importer and ONNX exporter, and proof-assistant export
(Lean/Coq/Isabelle obligation stubs). It is *not* a GPU-accelerated training
framework and it does *not* perform type-level / dependent-type shape proofs;
see <<Status and limitations>> and <<Prior art>> for an honest accounting of
where it sits relative to the established ecosystem.

The name and the older marketing line ("provably correct machine learning")
overstate what the code does today. The verification layer performs *runtime
property checks against supplied data*, plus optional SMT queries via the
bundled `packages/SMTLib.jl`. It is best understood as a runtime
invariant-checking layer for small Julia models, not a theorem prover for
networks.

== Installation

[source,julia]
----
using Pkg
Pkg.add("Axiom")
----

Requires Julia 1.10 or newer. All hard dependencies are registered Julia
packages (`HTTP`, `JSON`, `JSON3`, `Zygote`, `SHA`, plus standard libraries);
there are no native build steps for the default pure-Julia path. The optional
Zig backend and the CUDA/AMDGPU/Metal/PyCall package extensions are *not*
required for normal use.

== Minimal working example

The following runs end to end on a clean checkout
(`julia --project=. minimal.jl`). The output shown below is the verbatim
output produced when this example was run during documentation review.

[source,julia]
----
using Axiom

model = Sequential(
    Dense(784, 128, relu),
    Dense(128, 10),
    Softmax(),
)

x = Tensor(randn(Float32, 16, 784))
y = model(x)
println("output shape: ", size(y.data))

result = verify(model;
    properties = [ValidProbabilities(), FiniteOutput()],
    data = [(x, nothing)])
println("verification passed: ", result.passed)
println("properties checked: ", length(result.properties_checked))
----

Output:

[source]
----
output shape: (16, 10)
verification passed: true
properties checked: 2
----

`verify` evaluates the model on the supplied `data` and checks each property
against the actual outputs. `ValidProbabilities` checks that each row is
non-negative and sums to one; `FiniteOutput` checks the absence of `NaN`/`Inf`.
Both are *empirical checks on the data you pass in*, not universal proofs.

== Status and limitations

This is a young package. Be aware of the following before depending on it:

* *Verification is runtime, data-driven property checking*, not formal proof.
  `verify(model; properties, data)` runs the model on `data` and checks the
  properties on those outputs. `@prove` exists and the bundled SMTLib path
  loads, but trivial queries such as `@prove ∃x. x > 0` currently return a
  result of `unknown` with 0% confidence rather than a discharged proof. Treat
  `@prove` as experimental.
* *No "compile-time" type-level shape checking.* The `@axiom` DSL and the
  layer constructors raise `DimensionMismatch`/`ArgumentError` when a model is
  *constructed* or *run* with incompatible shapes — earlier than a deep
  training-time crash, but this is ordinary runtime/construction-time
  validation, not dependent-type verification.
* *The certificate workflow is currently broken.* `generate_certificate`
  references a `VerificationResult.mode` field that does not exist, so
  `generate_certificate` / `save_certificate` and the shipped
  `examples/simple_classifier.jl` fail at the certificate step. Do not rely on
  certificate generation until this is fixed.
* *GPU and coprocessor backends are stubs/extensions.* CUDA/AMDGPU/Metal are
  package extensions; the "TPU/NPU/DSP/FPGA/QPU" backends are type stubs with
  no real hardware path. `detect_coprocessor()` returns `nothing` on ordinary
  machines.
* *Test suite.* As of this writing the suite reports roughly 542 passing tests
  with ~100 errors concentrated in one property-based invariant
  ("Dense layer preserves batch size"). Core layer, serving, interop, proof
  serialisation, and invertible-layer testsets pass.

== Prior art

Axiom.jl overlaps with several mature, widely used Julia projects. An honest
comparison:

[cols="1,3,3",options="header"]
|===
| Project | What it does (and does better than Axiom.jl) | What Axiom.jl does differently

| https://fluxml.ai/[*Flux.jl*]
| The de-facto Julia deep-learning framework. Mature layers, training loop,
  GPU support, large ecosystem, battle-tested autodiff via Zygote. For
  building and training real models, Flux is strictly more capable and
  better supported.
| Axiom bundles a first-class `verify(model; properties, data)` step and
  property types (`ValidProbabilities`, `FiniteOutput`, `NoNaN`, `NoInf`) as a
  named API, plus an `@axiom`/`@ensure`/`@prove` DSL. Flux has no built-in
  property-verification concept; you would add such checks yourself.

| https://lux.csail.mit.edu/[*Lux.jl*]
| Explicit-parameter functional NN framework with strong composability,
  immutable parameters, excellent SciML integration and GPU/AD support. More
  principled and far more mature than Axiom's container model.
| Axiom uses conventional mutable layer structs and focuses on the
  post-inference invariant-checking and model-packaging/serving surface rather
  than parameter-management ergonomics.

| https://cuda.juliagpu.org/[*CUDA.jl*] /
  https://amdgpu.juliagpu.org/[*AMDGPU.jl*] /
  https://github.com/JuliaGPU/Metal.jl[*Metal.jl*]
| Production GPU array/kernel backends. Axiom.jl does *not* provide GPU
  acceleration of its own; these libraries are what actually run ML on GPUs in
  Julia and are vastly more capable on that axis.
| Axiom declares CUDA/AMDGPU/Metal only as optional package *extensions* and
  runs pure-Julia by default. Its differentiator is verification/packaging, not
  device performance.

| https://github.com/JuliaGPU/KernelAbstractions.jl[*KernelAbstractions.jl*]
| Vendor-agnostic GPU kernel authoring used across the JuliaGPU stack. A
  serious portable-kernel abstraction; Axiom has nothing comparable.
| Axiom's optional native path is a hand-written Zig FFI backend
  (`zig/src/*.zig`), loaded only when `AXIOM_ZIG_LIB` is set — a much narrower,
  optional escape hatch, not a kernel framework.

| https://github.com/EnzymeAD/Enzyme.jl[*Enzyme.jl*] /
  https://github.com/FluxML/Zygote.jl[*Zygote.jl*]
| State-of-the-art Julia automatic differentiation. Axiom does not implement
  AD; it *uses* Zygote and is strictly downstream of it.
| Axiom re-exports a thin `gradient`/`jacobian`/`pullback`/`GradientTape`
  wrapper over Zygote plus custom rules for its invertible layers. No novel AD.
|===

Summary: for training real models on real hardware, Flux.jl / Lux.jl with
CUDA.jl/AMDGPU.jl/Metal.jl and Enzyme.jl/Zygote.jl are the right tools and are
more mature on every performance and ecosystem axis. Axiom.jl's only genuine
differentiator is that property verification, model packaging/registry
manifests, and a multi-protocol inference-serving surface are first-class,
named APIs in one package. Whether that bundling is worth a dependency is a
judgement call; the verification it offers is runtime property checking, not
proof.

== Project structure

[source]
----
Axiom.jl/
├── src/
│   ├── Axiom.jl          # Main module + exports
│   ├── types/            # Tensor / Shape types
│   ├── layers/           # Dense, Conv2d, norm, pooling, invertible
│   ├── dsl/              # @axiom, @ensure, @prove macros
│   ├── autograd/         # Zygote wrappers
│   ├── training/         # Optimizers, losses, train!
│   ├── verification/     # Properties, checker, certificates (see caveat)
│   ├── backends/         # Julia backend + Zig FFI + GPU/coproc stubs
│   ├── serving/          # REST / GraphQL / gRPC
│   └── integrations/     # PyTorch import, ONNX export
├── zig/src/              # Optional hand-written Zig kernels
├── ext/                  # CUDA/AMDGPU/Metal/PyCall package extensions
├── packages/SMTLib.jl    # Bundled Julia SMT helper for @prove
├── test/                 # Test suite
└── docs/wiki/            # Documentation
----
