= Axiom.jl — Show Me The Receipts
Jonathan D.A. Jewell <j.d.a.jewell@open.ac.uk>
:toc:

The README makes claims. This file backs them up — or, where the code does not
support a claim, says so plainly. Every "verified" note below was checked by
actually running the code on a clean checkout during documentation review
(Julia 1.12, `julia --project=.`). Where a claim was found to be false or
stale, it has been downgraded here rather than dressed up.

== What it actually does today

=== Neural-network layers with correct forward passes

*Verified.* `Dense(in, out, activation)` (`src/layers/dense.jl`) runs a forward
pass with optional bias and activation and raises `DimensionMismatch` on wrong
input shapes. `Conv2d` (`src/layers/conv.jl`) computes correct output spatial
dimensions. `Sequential`/`Chain`/`Residual` containers work — a
`Sequential(Dense(784,128,relu), Dense(128,10), Softmax())` applied to a
`Tensor(randn(Float32,16,784))` produces a `(16,10)` output, verified by
running it. Activations `relu`, `sigmoid`, `tanh`, `softmax`, `gelu`,
`leaky_relu` and the capitalised layer wrappers (`ReLU`, `Softmax`, …) are
exported and exercised by the test suite (Dense/Conv2d/Activations/
Normalization/Pooling/Pipeline testsets all pass). Invertible layers
(`CouplingLayer`, `ActNorm`, `Invertible1x1Conv`, `RevBlock`,
`NormalizingFlow`) with `inverse`, `log_abs_det_jacobian`,
`forward_and_log_det` are present and covered by the "Invertible Layers"
testset (79/79 passing).

=== The `@axiom` DSL

*Implemented; not compile-time type verification.* `src/dsl/axiom_macro.jl`
implements `@axiom ModelName begin … end`. The README's `MNISTClassifier`
example (Conv → ReLU → MaxPool → … → Dense → Softmax) was run and constructs
successfully. `@ensure` (`src/dsl/ensure.jl`) works as a runtime assertion —
verified with both a standalone scalar predicate and inside `@axiom`.

*Caveat (unchanged, still accurate):* "compile-time" here means
macro-expansion / construction time, not Julia's type system. Shape mismatches
surface when the model is *built or run*, not when the file is parsed. This is
earlier than a deep training-time crash but is *not* dependent-type
verification. The README no longer claims otherwise.

=== `@prove`

*Downgraded — previous claim was misleading.* The earlier EXPLAINME said
"The `@prove ∃x. x > 0` example from the README works with the Julia-native
backend." Running it: `@prove ∃x. x > 0` executes without error but returns a
`ProofResult` of `unknown` with 0.0% confidence and a message that no
definitive proof or disproof was found. It does *not* discharge the proof.
`@prove` and the bundled `packages/SMTLib.jl` load and run, and the "SMT
Runner"/"SMT Cache" testsets pass (1 test each), but `@prove` should be
described as *experimental*, which the README now does. Claiming it "works" was
not accurate.

=== Verification properties, certificates, telemetry, proof export

*Mixed — one part is broken.*

* Properties (`src/verification/properties.jl`): `ValidProbabilities`,
  `FiniteOutput`, `NoNaN`, `NoInf` exist. *Verified:* `verify(model;
  properties=[ValidProbabilities(), FiniteOutput()], data=[(x, nothing)])`
  returns a `VerificationResult` with `passed == true` and two checked
  properties. These are empirical checks on the supplied data, not proofs.
* Telemetry (`src/verification/verification.jl`): *verified.*
  `reset_verification_telemetry!`, `verification_result_telemetry`,
  `verification_telemetry_report` run and produce a structured report.
* Certificates (`src/verification/certificates.jl`): *BROKEN.*
  `generate_certificate(model, result)` references `result.mode`, but
  `VerificationResult` has fields `passed`, `properties_checked`,
  `runtime_seconds`, `counterexamples`, `warnings` — no `mode`. Calling
  `generate_certificate` therefore throws `FieldError`. The shipped
  `examples/simple_classifier.jl` crashes at exactly this step. The README's
  old `verify(model) |> generate_certificate |> save_certificate` snippet has
  been *removed*; the README now lists this as a known bug. There are no
  certificate tests in `test/runtests.jl`, which is why this went unnoticed.
* Proof-assistant export (`src/proof_export.jl`): the "Proof Assistant Export
  Bundle" testset passes (30/30). Note `proof_obligation_manifest` takes a
  `ProofCertificate`, not a model — the earlier wording implying it operates
  on a model directly was imprecise; export is exercised via the certificate
  type in the tests, not via a bare model.

=== REST, GraphQL, and gRPC serving

*Verified.* `src/serving/api.jl` exports `serve_rest`, `serve_graphql`,
`serve_grpc`, `generate_grpc_proto`. The functions exist; `generate_grpc_proto`
was run and produces a `.proto` file. The "Serving APIs" testset passes
(42/42). HTTP is a hard dependency in `Project.toml`.

=== PyTorch import and ONNX export

*Verified.* `src/integrations/interop.jl` exports `from_pytorch` and `to_onnx`.
`to_onnx` on a small `Sequential` was run and produced an ONNX file.
`from_pytorch` with a canonical descriptor JSON works — the required format
string is `axiom.pytorch.sequential.v1` (an unknown format raises a clear
`ArgumentError`). The direct `.pt/.pth/.ckpt` checkpoint path is gated behind
the optional `PyCall` extension (`AxiomPyTorchExt`). The "Interop APIs" testset
passes (15/15).

=== Zig backend

*Present, optional, unverified here.* `src/backends/zig_ffi.jl` defines
`ZigBackend`/`init_zig_backend`; `__init__` in `src/Axiom.jl` loads the Zig
shared library only when `AXIOM_ZIG_LIB` is set, with a `@warn` fallback to
pure Julia. `zig/src/` contains `matmul.zig`, `conv.zig`, `norm.zig`,
`attention.zig`, `activations.zig`, `pool.zig`, `threading.zig`. The compiled
`.so` is not built or exercised by the default test path; treat the Zig backend
as an optional, unverified-in-CI accelerator.

=== GPU / coprocessor backends

*Stubs and extensions only.* `ext/` contains exactly four extensions:
`AxiomCUDAExt.jl`, `AxiomAMDGPUExt.jl`, `AxiomMetalExt.jl`,
`AxiomPyTorchExt.jl`. There are *no* coprocessor (TPU/NPU/DSP/FPGA/…)
extensions — the earlier claim of "9 coprocessor extensions" was false; those
backends are type stubs in `src/backends/julia_backend.jl` with no hardware
path. `detect_coprocessor()` returns `nothing` on an ordinary machine
(verified).

== Project.toml — corrected

The earlier EXPLAINME described "`AcceleratorGate`/`HTTP`/`SHA`/`JSON`/`JSON3`/
`Zygote`/`Libdl` hard dependencies, 13 weak dependencies". That is *stale and
wrong*. Current `Project.toml`:

* Hard deps: `Dates`, `HTTP`, `JSON`, `JSON3`, `Libdl`, `LinearAlgebra`,
  `Random`, `SHA`, `Serialization`, `Statistics`, `Zygote`. *No
  `AcceleratorGate`* — its selection logic was vendored into
  `src/vendored/AcceleratorGateVendored.jl` and the external dependency
  dropped (see git history).
* Weak deps: exactly *four* — `AMDGPU`, `CUDA`, `Metal`, `PyCall` — with one
  extension each. Not 13.
* `license = "MPL-2.0"`, `version = "1.0.0"`, uuid
  `bbd403f8-dcc5-405a-84eb-8de9d358675c`.

== Test suite — corrected

The earlier metadata claimed "tests-passing 283" / "completion 99%" /
"Release Candidate". Running `Pkg.test()`: approximately *542 passed, 0 failed,
100 errored*. The 100 errors are concentrated entirely in one property-based
invariant, "Invariant: Dense layer preserves batch size", under the
"Property-Based Tests" testset. All other testsets pass: Tensor Types, Dense,
Conv2d, Activations, Normalization, Pooling, Pipeline, Optimizers, Coprocessor
Backends (stubs), Loss, Data, Serving APIs (42), Interop APIs (15),
Verification (4), Ensure, SMT Runner/Cache, Invertible Layers (79), Proof
Serialization (41), Proof Assistant Export Bundle (30), E2E Pipeline (19), and
the other property invariants. The package is *not* a clean "283 passing /
release candidate"; describe it as a young package with one known failing
property test plus the certificate bug above.

== Module load order

The include order in `src/Axiom.jl` is, in sequence: types
(`tensor.jl`, `shapes.jl`) → layers (`abstract` → `dense` → `conv` →
`activations` → `normalization` → `pooling`) → DSL (`axiom_macro` → `ensure` →
`prove` → `pipeline`) → autograd (`gradient` → `tape`) → invertible
(`invertible.jl` → `invertible_rules.jl`) → training (`optimizers` → `loss` →
`train`) → verification (`properties` → `checker` → `certificates` →
`serialization` → `proof_export`) → vendored AcceleratorGate selection →
backends (`abstract` → `julia_backend` → `gpu_hooks` → `zig_ffi`) → model
metadata/packaging → utils → serving → integrations. Verified by reading
`src/Axiom.jl` directly.

== Removed / unverifiable claims

The previous EXPLAINME contained a "Dogfooded Across The Account" table
asserting integration with `ZeroProb.jl`, `SiliconCore.jl`,
`AcceleratorGate.jl`, `QuantumCircuit.jl`, `VeriSimDB`, `Ephapax`, and a
Hypatia CI scan. None of these are dependencies of this package
(`AcceleratorGate` was explicitly *removed* and vendored), none are exercised
by the test suite, and the claims could not be verified from this repository.
That table has been deleted rather than left as unsubstantiated cross-promotion
— it is exactly the kind of unverifiable assertion that should not appear in
package documentation.
